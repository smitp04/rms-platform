2. Authentication — Google SSO Shared Across All Tools

🔐
Single Sign-On via Google Workspace (@devxailabs.com)
One NextAuth.js session covers RMS, EMS, CSAT, CRM Views, and Employee Feedback. The session token carries user.id, system_role, and pod_id. RBAC permissions are resolved server-side on every API route.
🔑 Provider
NextAuth.js v5 with the Google OAuth2 provider. Domain restricted to @devxailabs.com. The hd (hosted domain) parameter blocks external Gmail accounts.
🎫 Session Storage
JWT strategy. Session verified on every server-side API route via requireSession(). No cookie store dependency — stateless and horizontally scalable.
🛡️ RBAC Roles
ADMIN (CEO/CTO/CFO) · POD_LEAD (team leads) · CSM (client success managers) · EMPLOYEE (view-only own data)
🔄 Auto-provisioning
New Google Workspace accounts are automatically provisioned as rms_employees records via n8n Google Admin webhook. No manual onboarding required.
Browser (Chrome) │ Google OAuth redirect → accounts.google.com │ ↓ authorization_code ↓ NextAuth.js Callback (/api/auth/callback/google) │ Verify hd=devxailabs.com │ Lookup/create rms_employees record │ Attach system_role + pod_id to JWT ↓ Signed JWT (stored in httpOnly cookie) │ ├──▶ Next.js Server Components: getServerSession() ├──▶ API Routes: requireSession() → throws 401 if invalid └──▶ Client Components: useSession() via SessionProvider

4. Database Architecture — One DB, Multiple Namespaces

All tools share a single PostgreSQL database. Tables are namespaced by prefix to allow clean joins across tools (e.g., rms_employees is the canonical employee source shared by all).

rms_
employees, pods, projects, accounts, allocations, sprints, audit_logs, pnl_snapshots, compute_costs
ems_
salary_history, leaves, promotions, performance_reviews, documents, skills (extended)
csat_
surveys, responses, nps_scores, health_checks, action_items
crm_
zoho_deals, contacts, activities, deal_sync_log (mirror cache of Zoho)
Key design decisions:
• All monetary values stored as integer cents — no floating point errors
• All PKs use UUID — no enumeration risk
• All tables carry created_at, updated_at, deleted_at — soft deletes everywhere
• rms_employees is the canonical employee record used across RMS, EMS, CSAT, and Feedback tools

🗄️ RDS PostgreSQL
Multi-AZ deployment. Automated backups (7-day retention). One database, namespaced tables for all tools. Connection pooling via Prisma connection pool.
⚡ ElastiCache Redis
Session token cache. Sprint data caching. TanStack Query also caches aggressively on the client, reducing server round-trips.
🚀 ECS Fargate
Serverless containers — no EC2 management. Auto-scales from 2 to 6 tasks based on CPU/memory. Rolling deploys from GitHub Actions pipeline.

8. RMS — Resource Management System Phase 1–5 Complete

Core Modules

📅 Allocation Gantt
Interactive sprint board. Drag-and-drop (dnd-kit). Per-sprint allocation percentages. 100% cap validation. Bulk copy across sprint ranges. Audit trail per cell.
💰 PnL Engine
Per-project profitability. Revenue vs. employee cost per sprint. Gross margin %. Red-flag detection. Revenue gated behind ADMIN role. Cached snapshots.
🏃 Sprint Engine
2-week bi-weekly sprints. Auto-generated for full year. Allocations keyed to sprint_id, not date ranges — supports per-sprint % changes.
📋 Audit Logs
Service-layer writes on every allocation create/update/delete. Stores full JSON snapshots (old + new state). Employee activity drawer shows history per person.
🏠 Pods
Team groupings with a designated lead. POD_LEAD role scopes Gantt read/write access to their pod members only. Editable by ADMIN.
📊 Dashboard
KPIs: active projects, avg allocation (last 6 sprints), available employees (>20% bandwidth), projects in red. Trend chart. Clickable cards navigate to filtered views.

9. EMS — Employee Management System Future Phase

EMS extends rms_employees as the canonical employee record. All EMS tables use the ems_ prefix and foreign-key into rms_employees.id.

Key Tables

ems_salary_history (employee_id, effective_date, ctc_cents, reason, approved_by) ems_leaves (employee_id, leave_type, from_date, to_date, status) ems_promotions (employee_id, old_role_id, new_role_id, effective_date, notes) ems_performance_reviews (employee_id, period, rating, reviewer_id, summary) ems_documents (employee_id, type [offer/appraisal/nda], s3_key, uploaded_at) ems_skills (employee_id, skill_id, level [1-5]) ← extends rms_skills
💼 Salary & Payroll
Salary history with effective dates. CTC stored as cents (integer). Syncs to rms_employees.salary_ctc_cents for PnL cost calculations.
🎯 Skills & Matching
Skill proficiency (1–5 levels). Searchable via GET /api/v1/ems/employees?skill=AI&min_level=3. Enables smart resource allocation suggestions.
📁 Document Vault
Signed URLs to S3 for offer letters, appraisals, NDAs. Accessible via ADMIN and the employee themselves. Enforced by Prisma + S3 IAM policies.
SSO Note: EMS uses the same Google SSO session. An employee accesses their own EMS profile at /ems/me. HR and ADMIN can access all profiles at /ems/employees. No separate login — one session, all tools.

10. CSAT — Client Satisfaction Future Phase

📬 Survey Delivery
NPS / CSAT surveys triggered by sprint completion or project milestones (via n8n). Delivered to client contacts via email. Responses stored in csat_responses.
📈 Project Health Scores
Computed score per project per sprint from survey responses. Displayed on the RMS project detail drawer. Alerts on health score drop below threshold.

Key Tables

csat_surveys (project_id, trigger_type [sprint_end/milestone], sent_at, status) csat_responses (survey_id, respondent_email, nps_score, rating, comment, submitted_at) csat_health_checks (project_id, sprint_id, computed_score, is_at_risk) csat_action_items (health_check_id, description, owner_id, due_date, status)
SSO Note: Internal CSAT dashboards (/csat) use the shared Google SSO session. External client respondents receive a public survey link — no SSO required. Response collection is anonymous by default.

11. CRM Views — Zoho Mirror Future Phase

Rather than rebuilding a CRM, devx reads Zoho CRM data via n8n webhooks + scheduled syncs, stores a local cache in crm_ tables, and surfaces read-only views inside the platform.

Zoho CRM ──── n8n Webhook (Closed Won) ──▶ POST /api/v1/webhooks/crm │ Creates rms_projects + rms_accounts Caches in crm_zoho_deals Zoho CRM ──── n8n Scheduled Sync (nightly) ──▶ POST /api/v1/webhooks/crm/sync │ Updates deal stages, contact info Marks stale deals as CANCELLED
🔄 Bi-directional Sync
Webhook for instant Closed Won trigger. Nightly n8n job for deal stage updates (e.g., Active → On Hold if Zoho changes). RMS is the downstream consumer, Zoho is the source of truth for deals.
👁️ CRM View Pages
Read-only pipeline view at /crm. Shows deal stage, value, contact. Links through to the RMS project when a deal is Closed Won. No write-back to Zoho from RMS.

12. Employee Feedback Ecosystem Future Phase

🔄 360° Reviews
Peer reviews, upward feedback, and self-assessments per review cycle. Stored in feedback_reviews. Aggregated into EMS performance records.
🎯 Goal Tracking
OKRs per employee per quarter. feedback_goals table. Manager can set, review, and close goals. Linked to performance review cycles.
🎖️ Recognition
Peer shoutouts and kudos. feedback_recognition table. Surfaced on employee profiles. Optional Slack notification via n8n workflow.
SSO Note: All feedback flows through the same Google SSO session. Employees submit peer reviews at /feedback. Anonymity is enforced at the API layer — feedback is aggregated before managers see it. HR and ADMIN see individual responses.
